Abertura ⏱ ~1 min
Mostrar o topo da apresentação — Hero com os 4 números em destaque.

Bom dia / Boa tarde a todos.

Vou apresentar o CantinaGO — um sistema de gestão de cantina escolar desenvolvido como projeto acadêmico aqui no SENAC.

A proposta é simples: alunos fazem pedidos pela plataforma, e a cantina gerencia tudo em tempo real — produtos, pedidos, status, horário de funcionamento.

Mas o que quero destacar hoje não é só o que o sistema faz, e sim como ele foi construído — as decisões técnicas, os padrões que seguimos e as boas práticas que aplicamos.

Em números: 175 testes unitários, zero vulnerabilidades nas dependências e uma nota de segurança 8,5 de 10 — que vou detalhar ao longo da apresentação.

2
Arquitetura em Camadas ⏱ ~2 min
Rolar até o diagrama de fluxo: Routes → Controllers → Services → Repositories → PostgreSQL.

A primeira decisão importante foi a arquitetura em camadas, inspirada no padrão Clean Architecture.

O fluxo de uma requisição começa nas Routes — que só definem URLs e aplicam middlewares. Vai para os Controllers, que recebem a requisição e delegam para o Service. O Service é onde mora toda a regra de negócio. E o Repository é a única camada que fala com o banco.

O benefício prático disso: se um dia precisarmos trocar o PostgreSQL por outro banco, mudamos só os Repositories — o resto do código nem sabe que o banco existe.

Além disso, usamos Value Objects — classes como Email, Id e Produto que encapsulam validação. Um e-mail inválido é rejeitado no momento em que o objeto é criado, muito antes de chegar ao banco.

E temos o Result Pattern: todo endpoint retorna a mesma estrutura JSON — sucesso, status HTTP, dados e mensagem. O frontend nunca tem surpresa sobre o formato da resposta.

Pausa natural — rolar para a próxima seção.

Tecnologias ⏱ ~2 min
Mostrar o grid de tecnologias — backend primeiro, depois frontend.

No backend: Node.js com Express 5 — versão mais recente, com suporte nativo a async/await nos middlewares.

Banco de dados PostgreSQL, acessado com queries parametrizadas — falo mais sobre isso na segurança.

Autenticação com JWT — JSON Web Tokens — com uma camada extra chamada token versioning que implementamos.

Senhas com bcrypt, padrão de mercado para hash com salt automático.

Segurança HTTP com Helmet e proteção contra abuso com express-rate-limit.

E-mails transacionais via API HTTP do Brevo — escolhemos HTTP em vez de SMTP porque provedores de cloud bloqueiam portas SMTP por padrão.

Documentação automática com Swagger / OpenAPI, testes com Vitest.

No frontend: React 19, Vite 8 e React Router v7 — tudo na versão mais atual disponível.

Deploy automático pelo GitHub: backend no Render, frontend no Vercel.

4
Segurança ⏱ ~3 min
Mostrar a tabela de segurança — percorrer linha a linha conforme vai falando.

Essa é a parte que mais nos dedicamos. Vou passar pelas proteções principais.

Primeiro, JWT com Token Versioning. Normalmente, quando você troca sua senha, o token antigo ainda funciona até expirar. Aqui não — cada usuário tem um número de versão salvo no banco. Quando a senha muda, esse número incrementa e qualquer token antigo é rejeitado na hora.

As senhas são armazenadas com bcrypt. Nunca em texto puro, nunca MD5 ou SHA simples — bcrypt com salt automático, resistente a ataques de rainbow table.

Para força bruta, temos duas camadas: rate limiting por rota — login aceita no máximo 10 tentativas em 15 minutos — e um lockout no banco de dados depois de 5 falhas consecutivas. Como fica no banco, se o servidor reiniciar o bloqueio continua.

Para SQL injection: todos os repositórios usam queries com parâmetros — nunca concatenamos strings para montar SQL. É uma das vulnerabilidades mais comuns e uma das mais fáceis de evitar.

Temos validação de propriedade: uma cantina não consegue editar produtos de outra. Verificamos se o ID do usuário autenticado bate com o dono do recurso em toda operação de escrita.

E os segredos — chave JWT, chave de API, senha do banco — nunca estão no código. São variáveis de ambiente. Se a JWT_SECRET não estiver definida, o servidor nem sobe.

Pausa — deixar a tabela visível por alguns segundos antes de rolar.
5
Arquitetura do Frontend ⏱ ~2 min
Mostrar os 6 cards do frontend e depois o grid dos 5 contextos.

O frontend é uma SPA — Single Page Application. O usuário baixa o app uma vez e toda navegação acontece sem recarregar a página.

Temos proteção de rotas por perfil: aluno e cantina têm interfaces completamente separadas. Se um aluno tentar acessar o painel da cantina, é redirecionado automaticamente — e vice-versa.

Usamos a Context API do React para estado global, sem precisar de Redux. São 5 contextos, cada um com responsabilidade única.

O mais interessante é o NotificacoesContext: ele conecta ao Server-Sent Events do backend, recebe eventos em tempo real, toca um som e salva até 30 notificações no localStorage. Se o usuário fechar e reabrir o app, as notificações ainda estão lá.

Temos modo claro e escuro com preferência salva entre sessões, um sistema de toast notifications para feedback não-bloqueante, e um ErrorBoundary que captura erros de renderização do React — em vez de tela branca, o usuário vê uma mensagem amigável com opção de recarregar.

No total são 12 páginas organizadas por perfil: públicas, de aluno e de cantina.

6
REST & Object Calisthenics ⏱ ~2 min
Mostrar as duas colunas lado a lado — REST (azul) e Object Calisthenics (roxo).

O projeto segue dois paradigmas que guiaram o design da API e a escrita do código.

O primeiro é REST. Não é só usar HTTP — é usar corretamente. As URLs identificam recursos, não ações. Os métodos HTTP têm semântica própria: GET lê, POST cria, PUT atualiza, DELETE remove. E os status HTTP são precisos: 201 para criação, 401 para não autenticado, 403 para sem permissão, 404 para não encontrado, 409 para conflito. Nunca retornamos 200 para um erro.

O segundo é Object Calisthenics — um conjunto de disciplinas de escrita de código. As três que mais aplicamos aqui:

Early return: se há uma validação que falha, lançamos uma exceção e saímos — sem blocos else aninhados que dificultam leitura.

Primitivos encapsulados: em vez de passar uma string de e-mail por todo o sistema, temos uma classe Email com validação embutida. Se chegou até o banco, é um e-mail válido.

Nomes sem abreviações: o código é lido muito mais do que escrito. validarCantinaAutenticada é mais claro do que valCantAuth — qualquer um entende sem precisar de comentário.

Endpoints ⏱ ~1 min
Mostrar o mapa de endpoints — apontar os cadeados enquanto fala.

Aqui o mapa das rotas. Rotas de autenticação e leitura de cardápio são públicas — qualquer pessoa acessa sem login.

Operações de escrita exigem JWT. Mas mais que isso: criar, editar ou deletar um produto exige que o usuário seja a cantina dona daquele produto. O cadeado com "dona" representa essa validação de propriedade.

Reservas são protegidas para usuários autenticados, e atualização de status é restrita à cantina do pedido.

O endpoint SSE é o canal de tempo real — quando a cantina atualiza um pedido, a notificação chega no app do aluno instantaneamente, sem polling.

Perguntas Prováveis
Por que usaram REST e não GraphQL?
REST é mais simples de implementar e de documentar com Swagger, e mais familiar para quem vai consumir a API. GraphQL faria sentido se tivéssemos consultas com muitos campos variáveis — o que não é o caso de uma cantina escolar.
Por que JWT e não sessões no servidor?
JWT é stateless — o servidor não precisa guardar sessões em memória ou banco. Isso facilita escalar e é o padrão para APIs consumidas por SPAs. A desvantagem de não poder invalidar o token foi resolvida com o token versioning que implementamos.
Como garantem que os testes cobrem o que importa?
Os testes mockam dependências externas — banco, e-mail, JWT — e testam a lógica de negócio isolada. Isso garante que as regras de autorização, validação e fluxo de autenticação estão corretas independentemente da infraestrutura.
Por que Context API e não Redux ou Zustand?
Para o tamanho do projeto, Context API é suficiente e não adiciona dependência externa. Redux faz sentido em apps muito maiores com estado complexo e profundo. Aqui cada contexto tem responsabilidade única e o estado é raso o suficiente.
O que é Object Calisthenics na prática?
É um conjunto de 9 regras de disciplina de código criadas por Jeff Bay. As que mais aplicamos aqui: um nível de indentação por método, sem else (early return), primitivos encapsulados em Value Objects, e nomes sem abreviações. O objetivo é forçar código mais legível e com responsabilidades bem definidas.
Por que Brevo para e-mail e não Gmail/SMTP?
Provedores de cloud como Render bloqueiam portas SMTP por padrão para evitar spam. O Brevo oferece uma API HTTP na porta 443 que funciona em qualquer ambiente. Além disso, e-mails transacionais têm melhor entregabilidade quando saem de um serviço dedicado.
