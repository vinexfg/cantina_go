Boas Práticas de Segurança

Implementações que protegem a API contra as principais vulnerabilidades do OWASP API Security Top 10.

Proteção Implementação Nível
JWT com Token Versioning Cada usuário possui um token_version no banco. Ao trocar a senha o token antigo é invalidado imediatamente, mesmo antes de expirar. Alto
Hash de Senhas (bcrypt) Senhas nunca são armazenadas em texto puro. bcrypt com salt automático (custo 10) — resistente a ataques de rainbow table. Alto
Rate Limiting em Camadas Global: 120 req/min. Login: 10 req/15min. Registro: 5 req/15min — protege contra força bruta e abuso de endpoints. Alto
Lockout Persistente Após 5 tentativas de login falhas, conta bloqueada por 15 minutos. Estado salvo no PostgreSQL — sobrevive a reinicializações do servidor. Alto
SQL Injection — Prevenção Todos os repositórios usam queries parametrizadas ($1, $2...) via pg. Nenhuma concatenação de string em SQL. Alto
Validação de Propriedade Uma cantina não pode editar produtos ou dados de outra. Verificação de usuario.id === recurso.cantina_id em todas as operações de escrita. Alto
Headers de Segurança (Helmet) Content-Security-Policy, X-Frame-Options, HSTS, X-XSS-Protection e outros headers configurados automaticamente. Alto
CORS Restrito API aceita requisições apenas do domínio do frontend (Vercel). Configurado via variável de ambiente CORS_ORIGIN. Alto
HTTP Semântico Correto 401 para não-autenticado, 403 para sem permissão, 404 para não encontrado, 409 para conflito — sem vazar informações indevidas. Médio
Verificação de E-mail Código de 6 dígitos enviado por e-mail. Login bloqueado até verificação — impede contas com e-mails falsos. Médio
Body Size Limit Corpo das requisições limitado a 50 KB — protege contra ataques de payload gigante que causam esgotamento de memória. Médio
Segredos via Variáveis de Ambiente JWT secret, chaves de API e credenciais do banco nunca commitados. Servidor falha na inicialização se JWT_SECRET estiver ausente. Alto

Boas Práticas Implementadas

Além de segurança, o projeto adota padrões de desenvolvimento que facilitam manutenção e evolução.

  • ✓
    Testes Unitários com Mocks 175 testes cobrindo services, controllers, middleware e repositórios. Dependências externas (banco, e-mail, JWT) isoladas com mocks — testes rápidos e determinísticos.
  • ✓
    Tratamento Centralizado de Erros Um único middleware ErrorMiddleware captura todas as exceções, mapeia para o status HTTP correto e formata a resposta. Controllers ficam limpos de try/catch repetitivo.
  • ✓
    Migrações Automáticas de Banco Tabelas criadas automaticamente no primeiro boot via migrations.js com CREATE TABLE IF NOT EXISTS. Deploy não exige SQL manual.
  • ✓
    Notificações em Tempo Real (SSE) Server-Sent Events nativos do Node.js para atualização instantânea de status de pedidos. Sem polling do cliente, sem dependência de serviço externo.
  • ✓
    E-mails Transacionais Assíncronos Envio de e-mail de verificação e status de pedido via Brevo HTTP API. Fire-and-forget — falha de e-mail não derruba a requisição principal.
  • ✓
    Documentação Interativa (Swagger) Todos os endpoints documentados com OpenAPI 3.0. Interface acessível em /api/docs, protegida por autenticação administrativa.
  • ✓
    Normalização de Dados na Entrada E-mails convertidos para minúsculas e sem espaços em todos os pontos de entrada. Evita duplicatas e inconsistências silenciosas no banco.
  • ✓
    Keep-Alive para Serviço em Produção Workflow GitHub Actions pinga o servidor a cada 14 minutos para evitar hibernação do plano gratuito do Render.
Interface
Arquitetura do Frontend

O frontend é uma SPA (Single Page Application) com arquitetura orientada a contextos e proteção de rotas por perfil de usuário.

🗺️

Rotas por Perfil

Componentes RotaAluno e RotaVendedor protegem as páginas. Aluno que tenta acessar o painel da cantina é redirecionado automaticamente, e vice-versa.

🔔

Notificações em Tempo Real

NotificacoesContext conecta ao SSE, reproduz um som e persiste até 30 notificações no localStorage. O aluno é avisado instantaneamente quando o pedido é confirmado ou cancelado.

🌗

Modo Claro / Escuro

ThemeContext alterna o tema com um clique e persiste a preferência no localStorage. Usa data-theme no elemento raiz para aplicar os estilos via CSS.

🍞

Toast Notifications

ToastContext exibe mensagens de feedback não-bloqueantes para o usuário — erros, sucessos e avisos — sem interromper o fluxo da tela.

🛡️

Error Boundary

Componente de classe ErrorBoundary captura erros de renderização do React que escapariam para tela branca. Exibe uma mensagem amigável com opção de recarregar.

📄

12 Páginas

Login, CardápioPublico, EsqueciSenha, ResetarSenha, VerificarEmail — para todos. Menu, Perfil, MeusPedidos — para alunos. Vendedor, Reservas, Histórico — para cantinas.

Paradigmas
REST & Object Calisthenics

Dois conjuntos de princípios que guiaram o design da API e a escrita do código.

API RESTful

  • ✓
    Recursos identificados por URI Cada entidade tem sua própria URI (/api/produtos, /api/reservas, /api/cantinas). Verbos nunca aparecem na URL.
  • ✓
    Uso correto dos métodos HTTP GET para leitura, POST para criação, PUT para atualização completa, PATCH para atualização parcial, DELETE para remoção — cada um com sua semântica.
  • ✓
    Status HTTP semânticos 200 OK, 201 Created, 401 Unauthenticated, 403 Forbidden, 404 Not Found, 409 Conflict, 422 Unprocessable Entity — nunca 200 para erros.
  • ✓
    Comunicação stateless O servidor não guarda estado de sessão. Cada requisição carrega o JWT com todas as informações necessárias para ser processada de forma independente.
  • ✓
    Respostas uniformes (Result Pattern) Todos os endpoints retornam a mesma estrutura JSON: { success, status, data, message }. Previsível para o frontend consumir.
  • ✓
    Paginação em coleções grandes Endpoints de listagem suportam ?page= e ?limit=, retornando metadados de paginação (total, totalPages).

Object Calisthenics

  • ✓
    Apenas um nível de indentação por método Métodos dos services e repositories extraem lógica para métodos auxiliares, mantendo cada função pequena e com responsabilidade única.
  • ✓
    Sem uso do else (early return) Validações lançam exceções ou retornam cedo — o fluxo principal do código fica no nível de indentação mais raso, sem blocos else aninhados.
  • ✓
    Primitivos encapsulados em Value Objects Email, Id e Preco são classes com validação embutida — um e-mail inválido nunca chega ao banco por ser rejeitado na criação do objeto.
  • ✓
    Classes pequenas e coesas Cada classe tem uma única responsabilidade: AuthService autentica, EmailService envia e-mails, UsuarioRepository persiste usuários.
  • ✓
    Sem abreviações Nomes completos e descritivos em todo o código: validarCantinaAutenticada, incrementTokenVersion, findByTokenVerificacao — sem siglas obscuras.
  • ✓
    Hierarquia de exceções tipadas Em vez de lançar Error genérico ou retornar códigos de erro, o projeto usa uma hierarquia de exceções (AppException → subtipos) — cada falha tem tipo e significado próprios.
